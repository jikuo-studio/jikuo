# JIKUO Document Registry

This directory contains the draft structured registry for JIKUO development
metadata.

The registry is introduced by `JIKUO-DOCREG-01` because the product task map,
capability registry, scenario chains, and mount rules have outgrown a single
Markdown source.

## Authority Model

- `docs/registry/*.yaml`: L0 draft product-level source of truth for structured entries.
- `.jikuo/project_context.yaml`: L1 project-local active subset and mount bindings.
- `docs/governance/jikuo_execution_mounts.md`: L2 human-curated mount recipes.
- README / task-map docs: L3 human-readable projections and explanations.

During DOCREG-01B1 these files are still `v0_draft`. Markdown projections remain
in place until the registry schema and shard contents are reviewed.

## Shards

- `registry_index.yaml`: shard list, authority metadata, anti-shards, and known
  external registries such as policy and insight indexes.
- `work_orders.yaml`: compact metadata for files under `docs/work_orders/`,
  including forward `implements_capabilities` edges extracted from each work
  order body.
- `capabilities.yaml`: observed `CAP-*` capability IDs, implementation status,
  metadata completeness status, and a transitional generated reference cache.
- `scenario_chains.yaml`: seed scenario-to-capability forward edges.
- `mount_sets.yaml`: seed named document mount sets.

Do not hand-maintain reverse edges such as `used_by` or `referenced_by`.
Those views should be generated by future tooling from forward references.

The current `referenced_by_transitional_cache` field in `capabilities.yaml` is a
temporary DOCREG migration aid. It preserves discovery provenance while the old
task-map table is being retired, but it is not a source-of-truth relationship
field and should move to a generated view in DOCREG-01C.
