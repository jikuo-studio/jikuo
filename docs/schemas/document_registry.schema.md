# JIKUO Document Registry Schema v0 Draft

> **Schema version**: `jikuo.registry.v0_draft`
> **Status**: Draft for DOCREG-01B1.

## Purpose

The document registry gives JIKUO a machine-readable map of work orders,
capabilities, scenario chains, and mount sets without deleting the existing
Markdown task map.

## Shared Fields

| Field | Required | Meaning |
|---|---|---|
| `schema_version` | yes | Must be `jikuo.registry.v0_draft` during the DOCREG-01 draft window. |
| `registry_id` | yes | Stable shard identifier. |
| `entries` | shard-dependent | Ordered list of registry entries. |

## Work Order Entry

Required fields:

- `id`: stable work-order id, usually `JIKUO-*`.
- `title`: human title from the work-order file.
- `path`: repository-relative path to the Markdown work order.
- `sprint`: sprint id.
- `status`: compact status text.
- `implements_capabilities`: forward references to `CAP-*` ids.
- `capability_extraction_status`: how `implements_capabilities` was derived.
- `required_mount_sets`: forward references to `MOUNT-*` ids.

`capability_extraction_status` values:

- `extracted_from_work_order_body`: one or more `CAP-*` ids were found in the
  work-order Markdown source and copied as forward edges.
- `no_capability_refs_found`: the work-order source currently contains no
  explicit `CAP-*` ids. This is a data-quality signal, not proof that the work
  order implements no capability.

## Capability Entry

Required fields:

- `id`: `CAP-*` id.
- `title`: capability title or generated review placeholder.
- `status`: implementation or planning status.
- `metadata_status`: registry metadata completeness status.
- `entry_point`: command, file, or `pending_review`.
- `write_effect`: declared write effect or `pending_review`.
- `approval`: declared approval boundary or `pending_review`.
- `referenced_by_transitional_cache`: repository-relative documents where the
  capability appears, generated during the DOCREG transition.
- `referenced_by_transitional_cache_truncated`: optional boolean used when the
  generated transition cache intentionally omits additional references.

`metadata_status` is intentionally separate from `status`:

- `status` describes the capability's product / implementation state.
- `metadata_status` describes whether the registry has enough structured
  metadata for that capability.

Allowed draft values:

- `complete`: metadata was migrated from the task map capability registry or
  equivalent source.
- `metadata_missing_in_task_map`: the `CAP-*` id was observed in the repository
  but did not have full task-map registry metadata during DOCREG extraction.

`referenced_by_transitional_cache` is an explicitly temporary reverse cache.
It is allowed only in DOCREG draft capability entries so maintainers can audit
where each capability id was discovered. It is not a source-of-truth
relationship field and must be generated by tooling, not hand-maintained. The
long-term generated-view target is DOCREG-01C.

If a generated cache omits additional references, the list must still contain
only repository-relative paths. Use `referenced_by_transitional_cache_truncated:
true` for that fact instead of adding sentinel strings to the path list.

## Scenario Chain Entry

Required fields:

- `id`: `SCENARIO-*` id.
- `title`: human title.
- `status`: `seed`, `draft`, `accepted`, `implemented`, or `archived`.
- `capability_refs`: forward references to capability ids.

## Mount Set Entry

Required fields:

- `id`: `MOUNT-*` id.
- `title`: human title.
- `status`: `seed`, `draft`, `accepted`, `implemented`, or `archived`.
- `documents`: list of repository-relative paths.

## Relationship Rule

Maintain forward edges only:

- work orders may reference capabilities and mount sets;
- scenario chains may reference capabilities;
- mount sets may reference documents.

Do not hand-maintain reverse fields such as `used_by`, `referenced_by`, or
`source_work_orders`. Future tooling should compute those views.

The only current exception is `referenced_by_transitional_cache` in
`capabilities.yaml`. It exists to preserve DOCREG discovery provenance
during the migration window and must not be copied into new shards or treated
as an authored graph edge.

## Registry Index External Registries

`registry_index.yaml` may declare `external_registries` for registries that
already existed before DOCREG and are governed by another subsystem.

Required fields:

- `id`: stable external registry id.
- `path`: repository-relative path.
- `authority_owner`: subsystem or contract that owns that registry.
- `layer`: L0/L1/L2/L3 relationship to the document registry model.

External registry entries make the layered registry aware of policy and insight
indexes without taking ownership of their formats.
