# SPRINT_050_WO-PER-JIKUO-DOCREG-01: Layered Document Registry

> **Status**: DOCREG-01A implemented; transitional authority remains for registry stabilization and next sequencing.
> **Date**: 2026-05-17
> **JIKUO layer**: documentation registry / governance documentation.
> **Business meaning**: JIKUO's task map, capability registry, scenario chains, and mount rules are now too large to manage safely as one Markdown document. JIKUO needs a registry-backed documentation model so future agents and users can find the right source of truth without reading thousands of lines by hand.

---

## 1. Transitional Authority

`docs/registry/registry_index.yaml` and the first registry shards now exist as `v0_draft` DOCREG-01A artifacts. Until those shards are reviewed and promoted beyond draft, this work order remains the transitional source of truth for DOCREG sequencing and boundaries.

This document is intentionally a work order, not a permanent registry. It exists to prevent the registry migration itself from being hidden inside the oversized task map that it is meant to replace.

Authority during the transition:

- this work order: current DOCREG task sequence and accepted boundaries;
- `.jikuo/project_context.yaml`: project-local mount declaration for this work order;
- `docs/governance/jikuo_productization_task_map.md`: human-facing projection and summary pointer;
- `docs/registry/*.yaml`: draft machine-readable source of truth introduced by DOCREG-01A.

Projection edit rule:

- The task map is not the default write target for new task sequencing, open-item facts, capability metadata, or registry authority.
- During the DOCREG transition, edit the task map only for explicit projection notes, link repairs, frozen-section notices, or approved regeneration.
- If a future slice changes the task map outside those projection-maintenance cases, it must record why the registry / work-order authority was insufficient.

---

## 2. Accepted Direction

The current design direction is a four-layer model:

| Layer | Role | Meaning |
|---|---|---|
| L0 | `docs/registry/*.yaml` | Product-level source of truth for registry entries, relationships, and authority metadata. |
| L1 | `.jikuo/project_context.yaml` | Project-local active subset and mount bindings. |
| L2 | `docs/governance/jikuo_execution_mounts.md` | Human-curated mount recipes for task types and work contexts. |
| L3 | `README.md`, `docs/README.md`, `docs/governance/jikuo_productization_task_map.md` | Human-readable projections and orientation docs. |

Key decisions accepted for planning:

- registry files are the future source of truth, while long Markdown files become projections or narrative guides;
- `open_items.yaml` must not be a source shard because open items are a computed view over work-order and capability statuses;
- `mount_sets.yaml` is the preferred name for reusable document mount sets;
- reverse edges such as `used_by` or `referenced_by` should be generated by tools, not hand-maintained;
- DOCREG schemas stay `v0_draft` until reviewed after real data is migrated.
- task-map changes are projection maintenance by default, not source-of-truth updates.

## 3.1 Held Policy-Governance Candidates

These are user-proposed policy candidates captured during the policy-trigger and
work-profile modeling discussion on 2026-05-17. They are held here because this
work order is the current transitional task-sequencing authority. They are not
active policies, and they must not be written into `.jikuo/policies/approved`
until the user explicitly approves a policy-write plan.

Activation is intentionally deferred until the minimum architecture correction
around `work_profile`, lightweight `task_start`, and future evaluator boundaries
is accepted. The reason is practical: these candidates are about reasoning and
modeling behavior, so activating them before the policy-trigger model is stable
would create noisy governance instead of useful protection.

### POLICY-CANDIDATE-first-principles-critical-alignment

User-facing intent:

When the user proposes a requirement, architecture direction, policy, workflow,
or implementation detail, the assistant must not behave like an echo chamber.
It should reason from first principles, test the proposal against the underlying
business goal, and align concepts before accepting implementation details.

Internal reasoning requirement:

- Use 5 Whys as the default internal root-cause method.
- Stop before five iterations only when the root cause, target failure mode, and
  decision boundary are already clear.
- Expose the process lightly to the user. Usually 2-3 visible why questions or
  a concise root-cause summary are enough; the full internal chain does not need
  to be shown unless it helps the user decide.

Required distinction:

- user principle / desired outcome;
- user-proposed solution;
- root problem or failure mode;
- alternative implementation paths;
- recommended next step and residual risk.

Business meaning:

This protects users who can perceive product and governance needs but may not
know the best engineering shape. JIKUO should help them avoid accidentally
turning a provisional solution into durable architecture.

### POLICY-CANDIDATE-data-model-drift-alarm

User-facing intent:

When fields, schemas, layers, registries, classifications, or event types keep
growing and their relationships become blurry, the assistant must treat that as
a data-modeling risk, not as a cue to add another field.

Trigger signals:

- repeated questions like "what is the relationship between these fields";
- overlapping concepts such as event / lifecycle / task / task-session / scope;
- new field additions that do not clarify the business chain;
- source-of-truth, projection, derived view, and transitional cache becoming
  mixed in the same layer.

Required action:

- Reconstruct the real business chain before adding structure.
- Name source of truth, projection, derived view, and transitional cache.
- Decide whether the fix is a small field addition or a model-boundary rebuild.
- Prefer early architecture correction over late dashboard / BI cleanup.

Business meaning:

JIKUO is expected to support runtime cards, future dashboards, BI-style
analysis, and bug reports. Those features require clean facts. If the data
model tilts away from the actual user-work chain, later analytics will inherit
the confusion.

## 9. DOCREG-01B1 Implementation Record

DOCREG-01B1 is a registry semantics repair slice. It updates draft registry
data and documentation so the next hardening slice can enforce meaningful
checks instead of freezing known migration debt.

Scenario preserved from review:

- A maintainer or future agent reads `capabilities.yaml` and must not confuse a
  reverse discovery cache with the source-of-truth graph.
- A maintainer reads `work_orders.yaml` and needs to see which capabilities a
  work order explicitly mentions, rather than 90 empty edge lists.
- A user asks whether a capability is implemented; registry metadata debt must
  not overwrite the capability's implementation `status`.
- A future completion-review policy should be able to say "registry consistency
  was checked" with machine-test evidence, not only prose.

Implemented decisions:

- `capabilities.yaml` keeps `status` for product / implementation state and adds
  `metadata_status` for registry completeness.
- `source_refs` was renamed to `referenced_by_transitional_cache` and documented
  as a generated transition cache, not a hand-authored relationship edge.
- cache truncation is represented by
  `referenced_by_transitional_cache_truncated: true`, keeping cache list values
  as repository-relative paths.
- `work_orders.yaml` now carries `implements_capabilities` edges extracted from
  the source Markdown body plus `capability_extraction_status`.
- `registry_index.yaml` now knows about existing policy and insight registries
  without taking ownership of their formats.
- task map section 3 is frozen as a Markdown projection; capability metadata
  changes now belong in `docs/registry/capabilities.yaml`.

Non-goals:

- no runtime behavior changes;
- no generated projection tooling;
- no new registry shard;
- no hard-fail upgrade for all missing `CAP-*` metadata yet;
- no deletion of the task map capability table before DOCREG-01C.

Verification:

- `python -B -m unittest tests.document_registry_tests` passed with 7 tests.
- `python -B -m unittest tests.execution_events_schema_tests` passed with 5 tests.
- `python -B -m unittest discover -s tests -p "*_tests.py"` passed with 202 tests.

## 10. DOCREG-01B2 Implementation Record

DOCREG-01B2 hardens structural checks after B1 made the draft registry data
semantically meaningful.

Business scenario preserved:

- A maintainer adds a new work-order file and must register it before the slice
  can be considered documentation-governed.
- A maintainer adds a new insight file and must keep the insight registry in
  sync.
- A maintainer adds a new registry shard and must declare it in
  `registry_index.yaml`.
- A future agent must not create `open_items.yaml` as a source shard, because
  open items are a computed view over work-order and capability status.
- A future agent must not reintroduce hand-maintained reverse edges in registry
  shards.

Implemented hard checks:

- every `docs/registry/*.yaml` shard except `registry_index.yaml` is declared by
  the index;
- `docs/registry/open_items.yaml` must not exist;
- registered work-order, insight, mount, and external-registry paths must exist;
- every `docs/work_orders/*.md` file is registered in `work_orders.yaml`;
- every `docs/insights/INSIGHT-*.md` file is registered in
  `docs/insights/insights_registry.yaml`;
- reverse-edge fields such as `used_by`, `referenced_by`,
  `source_work_orders`, and `source_refs` are forbidden in registry shards;
- `capability_extraction_status` is a checked enum and must agree with whether
  `implements_capabilities` is empty.

Explicitly still warning-only:

- whole-repository `CAP-*` completeness. The current 16 capabilities marked
  `metadata_status: "metadata_missing_in_task_map"` are known migration debt.
  Future hardening should add an explicit grandfathering rule before turning
  all metadata completeness into a hard failure.

Stop point:

- DOCREG-01B3 is policy governance. It should add completion-review policy
  surfacing for registry maintenance, but it is intentionally not implemented
  in this B2 slice.

## 10.1 DOCREG-01B2a Projection Guard Note

Recorded on 2026-05-17 after a self-bootstrap review found that the oversized
task map was still easy to treat as a live task source.

Root cause:

- only the capability registry section was explicitly frozen;
- `.jikuo/project_context.yaml` still described the task map as the current
  todo map;
- `jikuo_execution_mounts.md` still told agents to check the task map for
  current snapshot / capability / todo changes;
- no hard check yet distinguishes projection maintenance from source-of-truth
  task sequencing.

Immediate repair:

- `project_context.yaml` now labels the task map as a legacy L3 projection;
- `jikuo_execution_mounts.md` now says the task map is mounted as projection
  context only;
- the task map header now warns that new task sequencing, open-item facts,
  capability metadata, and registry authority belong in DOCREG / registry
  sources instead.

Future hardening candidate:

- Add a registry / completion-review check that warns when
  `docs/governance/jikuo_productization_task_map.md` changes outside explicit
  projection repair, link fix, frozen-section notice, or approved regeneration.
