# SPRINT_050_WO-PER-JIKUO-LIVE-13: Taskmap Insight Follow-up Evidence

> **Status**: Accepted on 2026-05-14
> **Product meaning**: make the taskmap / insight / follow-up distinction auditable as structured JIKUO evidence instead of relying only on final assistant wording.
> **Scope rule**: add report-only work-routing evidence to the local runner; do not implement MCP, persistent task-session evidence writes, UI, dashboard, or gates in this slice.

## 1. Why This Slice Now

`POLICY-jikuo-taskmap-insight-followup-distinction` requires evidence that user-facing summaries distinguish taskmap work, insights, and follow-up / deferred items.

Before this slice, `agent_flow.py propose --event task_start` could trigger that policy but could not produce matching evidence. The result was a persistent missing-evidence report even when the assistant verbally distinguished the categories later.

## 2. User Scenario

- A JIKUO-governed task starts.
- The runner classifies the work as taskmap, insight, follow-up, policy, deferred, or mixed.
- The runner records this classification as `taskmap_insight_followup_distinction_evidence`.
- The policy evaluator can match the evidence through the existing report-only evidence checker.
- The chat-ready output shows a `Work Routing Evidence` section so the user can see the classification before final delivery.

## 3. Implementation Delivered

- `src/jikuo/agent_flow.py` now emits a `work_routing` object for task-start proposals.
- `src/jikuo/agent_flow.py` now produces `taskmap_insight_followup_distinction_evidence` for `distinguish_taskmap_insights_followups_in_user_summary`.
- `agent_flow.py propose` supports `--work-routing-category` and `--work-routing-summary` for explicit overrides.
- The default task-start route is `taskmap`, with separate empty buckets for insights and follow-ups.
- `chat_ready_markdown` renders `## Work Routing Evidence`.
- `tests/agent_flow_tests.py` verifies that a real policy-store fixture can satisfy `POLICY-jikuo-taskmap-insight-followup-distinction` without missing evidence.

## 4. Boundaries

- This is report-only evidence generated by the runner.
- It does not persist task-session evidence snapshots.
- It does not decide whether a user idea should become a starter policy template.
- It does not replace final-response discipline; the final response still needs to distinguish taskmap, insights, and follow-ups for the user.

## 5. Acceptance Criteria

- A task-start proposal can include structured `work_routing`.
- The policy evaluator can satisfy `taskmap_insight_followup_distinction_evidence`.
- Missing evidence for `POLICY-jikuo-taskmap-insight-followup-distinction` disappears in a representative task-start run.
- The chat-ready card shows the classification visibly.
- Existing no-write boundaries remain unchanged.
